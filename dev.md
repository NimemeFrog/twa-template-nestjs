# Сервер

## 🔐 Аутентификация (Telegram Web App)

В проекте реализована гибридная система аутентификации, специально адаптированная для Telegram Web App (TWA). Мы используем **WebAppData валидацию (HMAC-SHA-256) + JWT (JSON Web Tokens)**. 

Использование классических cookie-сессий здесь не применяется, так как TWA открывается в iframe внутри Telegram, что часто вызывает конфликты с кросс-доменными политиками (CORS) и блокировкой third-party cookies на мобильных устройствах.

### 🔄 Жизненный цикл аутентификации (Flow)

1. **Инициализация (Client):** При открытии Mini App, TWA SDK предоставляет строку `WebApp.initData`. Она содержит данные пользователя, `auth_date` и криптографический `hash`.
2. **Вход (Client -> Server):** Клиент отправляет сырую строку `initData` на бэкенд (`POST /api/auth/telegram-login`). Мы **не используем** `initDataUnsafe`, так как его легко подделать.
3. **Валидация (Server):** - Бэкенд проверяет параметр `auth_date`. Если данным больше 5 минут — запрос отклоняется (защита от Replay Attack).
   - Формируется строка проверки данных (Data-check-string) согласно документации Telegram.
   - С помощью `BOT_TOKEN` и алгоритма `HMAC-SHA-256` вычисляется серверный хэш.
   - Вычисленный хэш сравнивается с хэшем из `initData`. Если они совпадают — 100% гарантия, что запрос пришел от реального пользователя через Telegram.
4. **БД и Токен (Server):** При успешной проверке бэкенд ищет пользователя в базе (Prisma) по `telegramId` или создает нового. После этого генерируется и возвращается клиенту `accessToken` (JWT).
5. **Защищенные API (Client -> Server):** Клиент сохраняет токен и передает его в заголовке `Authorization: Bearer <token>` ко всем закрытым API.
6. **Авторизация (Server):** В NestJS используется `JwtAuthGuard`, который математически валидирует токен на подлинность и срок действия без дополнительных запросов к базе данных (Stateless).

### ⚙️ Переменные окружения (.env)
Для корректной работы модуля авторизации обязательны:
* `BOT_TOKEN` — токен вашего бота от @BotFather. Используется как соль для проверки подлинности данных от Telegram.
* `JWT_SECRET` — длинная и надежная случайная строка. Используется для подписи выпускаемых JWT токенов бэкендом.

### 📁 Основные файлы модуля
* `src/api/auth/auth.module.ts` — регистрация модуля и настроек JWT.
* `src/api/auth/auth.service.ts` — ядро бизнес-логики: валидация подписи Telegram, генерация токенов, работа с Prisma.
* `src/api/auth/auth.controller.ts` — публичный эндпоинт авторизации.
* `src/api/auth/jwt-auth.guard.ts` — Guard для закрытия API эндпоинтов (используется как `@UseGuards(JwtAuthGuard)` в контроллерах).

> ⚠️ **Важно для разработчиков фронтенда:** При тестировании TWA в обычном браузере (вне Telegram), строка `initData` будет пустой. Тестировать авторизацию нужно либо через локальный туннель (ngrok/localtunnel) прямо в клиенте Telegram, либо захардкодив валидную (свежую) строку `initData` для локальной разработки.

## Разработка
Для запуска клиента (во время разработки): ssh -R 80:localhost:5173 nokey@localhost.run
Для запуска бекенда (во время разработки): ssh -R 80:localhost:3000 nokey@localhost.run
